You are an expert full-stack developer. Build a production-ready data collection application for the Saiban Orphan Support Program.

PROJECT GOAL
Create a temporary data collection app for collecting orphan registration data from field workers. Later this data will be migrated into the main Saiban application, which uses PostgreSQL. So the database schema must be clean, relational, and migration-friendly.

TECH STACK
- Next.js App Router
- TypeScript
- Tailwind CSS
- Neon PostgreSQL
- Cloudflare R2 for media/documents storage
- Prisma ORM or Drizzle ORM, choose the better option and set it up cleanly
- Zod for validation
- Server Actions or API routes where appropriate

IMPORTANT CONTEXT
The main Saiban app already has concepts like:
- Family
- Individual
- OrphanProfile
- Assessment
- AssessmentDocument

The temporary data collection app should use similar structure so migration becomes easy later.

- father_death_certificate
- mother_cnic
- guardian_cnic
- school_letter
- imam_verification
- principal_verification
- house_photo

Upload rules:
- Images must be compressed before upload if larger than 1MB.
- Final image size should be around or below 1MB.
- Allowed image types: jpg, jpeg, png, webp.
- Convert images to webp where suitable.
- PDFs are allowed for documents.
- Store only file_key, file_url if needed, mime_type, size, document_type in database.
- Prefer storing file_key as main reference because CDN/domain can change later.

5. Database Design
Use separate temporary database tables but migration-friendly names.

Suggested tables:
- users
- orphan_applications
- siblings
- relatives
- household_assets
- application_documents
- audit_logs

Every main table should include:
- id UUID primary key
- migration_status: pending, validated, migrated, rejected
- main_saiban_id nullable
- migration_errors nullable
- created_by
- created_at
- updated_at

Application status:
- draft
- submitted
- needs_correction
- validated
- rejected
- migrated

6. Validation
Use Zod validation.

Rules:
- CNIC must be 13 digits after removing dashes.
- B-Form must be 13 or 15 digits depending on current form requirement; keep validation configurable.
- Date of birth cannot be future.
- Father date of death cannot be future.
- Child age at registration must be less than 12 years.
- Monthly income, rent, fee, and expenses must be non-negative.
- Latitude must be between -90 and 90.
- Longitude must be between -180 and 180.
- Required documents must be checked before final submission.

7. Urdu + English Labels
Create a central labels config file:

formLabels.ts

Example:
{
  mother_name: {
    en: "Mother Name",
    ur: "والدہ کا نام"
  }
}

Use this config throughout the UI.
Do not hardcode labels everywhere.

10. Security
- Do not expose R2 secret keys to frontend
- Use signed upload URLs or backend upload route
- Validate file type and size on backend too

- Use environment variables

Required environment variables:
- DATABASE_URL
- R2_ACCOUNT_ID
- R2_ACCESS_KEY_ID
- R2_SECRET_ACCESS_KEY
- R2_BUCKET_NAME
- R2_PUBLIC_URL or CDN_URL
- NEXTAUTH_SECRET or AUTH_SECRET
